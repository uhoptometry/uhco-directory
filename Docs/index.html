1. Overview

UHCO Ident is the authoritative directory application for UHCO faculty, staff, current students, and alumni. It centralizes roster management, syncs with the university-wide UH Directory API, and exposes its own secured REST API for downstream systems such as Modern Campus.

Technology Stack

  • Back end: Adobe ColdFusion, developed on CF2025 and deployed on CF2021
  • Database: Microsoft SQL Server via datasource UHCO_Directory
  • Front end: Bootstrap 5 and Bootstrap Icons in the admin interface
  • Web server: IIS with URL Rewrite for API routing
  • External integration: UH Directory API with Bearer token and HMAC secret
  • Self-service workflow: UserReview staged profile updates with admin approval

Application URL Structure

  • /admin/ administrative back-office
  • /UserReview/ end-user self-service review experience
  • /api/v1/ REST API endpoints
  • /cfc/ service components
  • /dao/ DAO components
  • /includes/ shared helper partials
  • /sql/ schema and migration scripts
  • /xml/ XML data used by the UH API integration

2. Core Data Model

All data is stored in SQL Server and hangs off the central Users table. The model is effectively a star pattern with user identity at the center and related records branching by concern.

3. User Management

The admin/users/ area is the core administrative workspace for managing directory records. The app treats five user groupings as first-class admin views: Faculty, Staff, Current Students, Alumni, and Inactive.

User List

  • Paginated, sortable table with filters driven by tabs and query-string state.
  • Dedicated “Needs Attention” view for records carrying flags like Admin - Check or No UH.
  • Bulk flag and organization lookups are loaded up front to avoid N+1 queries.
  • Sort links are generated by helpers and the active sort column is highlighted.

User Profile View

The profile view uses directory_service.getFullProfile(userID) and renders identity, contact info, titles, photo gallery, organizations, flags, external IDs, addresses, academic info, sync history, and data-quality exclusions.

Create, Edit, Delete

  • new.cfm creates a blank user record.
  • edit.cfm loads the same save flow with existing data.
  • saveUser.cfm validates required fields and email format before writing via users_service.saveUser().
  • Delete flows hard-delete the user and rely on related table cleanup to remove dependent rows.

Per-User UH API Sync

Admins can sync a single user from the UH Directory API and resolve field-by-field differences, or use a lightweight AJAX endpoint to refresh a single field directly from the upstream source.

4. UserReview Workflow

UserReview is a separate self-service flow that allows eligible users to stage profile changes without writing directly to the live Users record. Changes remain pending until an authorized admin approves or rejects them.

User Experience

  • Public entry point lives under /UserReview/ with its own login and session namespace.
  • Eligibility is based on feature settings plus user flags such as Faculty and Staff, not the admin-role system.
  • Title1 and EmailPrimary are shown read-only for reference and cannot be edited.
  • Existing secondary emails, phones, and addresses are preloaded to prevent duplicate entry.
  • Plain @uh.edu addresses are excluded from editable secondary email rows; @central.uh.edu and @cougarnet.uh.edu are allowed.

Submission Lifecycle

  1. User loads the form model from live profile data plus any unresolved pending changes.
  2. Only actual diffs are stored; unchanged contact collections are ignored by canonical structural comparison.
  3. Submission data is written to UserReviewSubmissions and UserReviewSubmissionFields.
  4. Admins may approve or discard an entire submission or resolve individual fields.
  5. Approved fields update live data immediately; discarded fields retain the current value.
  6. Submission status becomes approved, rejected, or partially_approved after all fields are resolved.

Review Notes

Discard actions require a reason. Notes are appended to UserReviewSubmissions.ReviewNote, shown on the admin review page, and surfaced back to the user when the most recent reviewed outcome is rejected or partially_approved. If the user creates a new pending submission, the prior review-update alert is suppressed until the new submission is reviewed.

Key Files

  • UserReview/index.cfm, login.cfm, authenticate.cfm, save.cfm, layout.cfm
  • model/services/UserReviewAuthService.cfc for end-user authentication and session state
  • model/services/userReview_service.cfc for form modeling, diffing, and approval logic
  • model/dao/UserReview_DAO.cfc for submission persistence and review-state queries
  • sql/create_user_review.sql for schema and default app settings

5. Organizations

Organizations provide the hierarchy used for browsing, filtering, and representing the college’s internal structure.

Structure

  • OrgID, OrgName, OrgAbbr
  • ParentOrgID supports a self-referential tree
  • OrgType controls display grouping
  • SortOrder controls sequence
  • Active hides retired orgs from display and API output

User Assignments

UserOrganizations allows multiple org memberships per user with start and end dates. Admin pages support add/remove membership management directly from the user profile.

6. Flags

Flags are used for filtering, admin workflows, and API data protection.

Definition Model

  • FlagID, FlagName, FlagDescription
  • Protected marks flags that require a valid secret to be returned in API responses
  • SortOrder and Active control display and retirement behavior
Protected data behavior: flags such as Current-Student and Alumni are filtered from API responses unless the caller presents a valid secret that unlocks those protected flags.

7. REST API

The API is rooted at /api/v1/ and is read-only. All requests require a valid Bearer token, and some protected data additionally requires a secret.

For full endpoint reference, authentication examples, quickpull behavior, and response samples, see the dedicated API documentation at /api/docs.html.

Key Endpoints

GET /api/v1/people
GET /api/v1/people/{id}
GET /api/v1/people/{id}/flags
GET /api/v1/people/{id}/organizations
GET /api/v1/people/{id}/academic
GET /api/v1/people/{id}/addresses
GET /api/v1/people/{id}/externalids
GET /api/v1/organizations
GET /api/v1/organizations/{id}
GET /api/v1/flags

People Query Parameters

  • search full-text search across identity and email fields
  • flag filter by flag name
  • org filter by organization ID
  • class filter by academic cohort year
  • limit max 500, default 50
  • offset pagination offset
  • format json or xml

Token Validation Pipeline

  1. Authorization header is present and correctly formatted.
  2. SHA-256 hash of the token matches a stored record.
  3. Token is not revoked.
  4. Token is not expired.
  5. Caller IP satisfies allowlist or CIDR rules.
  6. Requested scope is permitted.
  7. LastUsedAt is updated on success.

Error Format

{ "error": "message", "status": 401 }

8. Tokens and Secrets

API tokens and secrets are managed separately because they solve different security problems.

Tokens

  • Grant API access
  • Store only SHA-256 hashes, never the raw token
  • Track scope, allowlist, expiration, revocation, and last used time
  • Can be revoked or permanently deleted

Secrets

  • Unlock protected flag and organization data
  • Also store only SHA-256 hashes
  • Define which protected flags a client may see
  • Use the same one-time reveal lifecycle as tokens

9. UH Directory Sync

UHCO Ident maintains its own copy of staff and faculty data and periodically compares it against the upstream UH Directory API.

Credentials and Configuration

  • UH_API_TOKEN Bearer token
  • UH_API_SECRET HMAC signing secret
Important: the source documentation notes hardcoded fallback credentials in admin/reporting/run_uh_sync_report.cfm. Those should be removed and replaced with environment-only configuration.

Bulk Sync Flow

  1. Load all local users with a non-null UH_API_ID.
  2. Retrieve current staff and faculty from the UH API.
  3. Build a lookup keyed by UH_API_ID.
  4. Compare each local record field by field.
  5. Write differences to UHSyncDiffs.
  6. Track local-only records in UHSyncGone.
  7. Track upstream-only people in UHSyncNew.
  8. Create a UHSyncRun record with counts and metadata.

10. Data Quality

The data-quality system audits the local database for missing or inconsistent information and stores findings for review.

Typical Issue Codes

NO_EMAIL NO_PHONE NO_ROOM NO_BUILDING NO_TITLE NO_DEPT NO_ORG NO_FLAG NO_PHOTO NO_UH_ID DUP_EMAIL DUP_UH_ID INACTIVE_ORG INACTIVE_FLAG NO_ACADEMIC NO_GRAD_YEAR MISSING_PREFERRED

Excluded issues can be suppressed per user via DataQualityExclusions when a known exception is acceptable.

11. Bulk Tools and Reporting

The admin reporting area contains recurring import, audit, and migration utilities beyond sync and DQ.

  • CS Bulk Import: mass-creates new College of Science users with initial org and flag assignments.
  • CS Alumni Bulk Import: creates alumni records with graduation metadata.
  • CS Migration Tool: one-time legacy data migration utility.
  • OD Student Audit: compares the student roster against enrollment-related data.
  • UH People Import: interactive import from the UH API.
  • UH People — DB Not In API: finds local records whose UH_API_ID no longer appears upstream.

12. Admin Interface

The admin side is a server-rendered Bootstrap application with shared layout, dashboard, reporting pages, and operational utilities.

Highlights

  • dashboard.cfm shows DQ and sync status badges plus quick navigation.
  • layout.cfm provides the shell, sidebar, current user display, and navigation.
  • diagnostic.cfm surfaces runtime environment and connectivity information.
  • clear_cache.cfm and resetapp.cfm are operational maintenance pages.
  • admin/access/ manages named access areas.
  • admin/external/ manages external system definitions.
  • admin/settings/user-review/ combines UserReview configuration and submission processing.

Permission Model

  • settings.user_review.manage controls UserReview feature configuration.
  • users.approve_user_review controls submission review, approval, and rejection.
  • The UserReview admin landing page can show configuration, submissions, or both depending on the caller's granted permissions.
  • Review actions refresh the admin authorization context before writing approval metadata so stale sessions do not resolve fields with AdminUserID = 0.

13. Architecture and Code Structure

UHCO Ident uses a simple three-layer design with request-time object creation and no dependency injection framework.

Layer Overview

  • Presentation layer: .cfm pages in admin/ and api/v1/
  • Service layer: CFCs in cfc/
  • Data access layer: CFCs in dao/, all extending BaseDAO.cfc

Instantiation Pattern

<cfset svc = createObject("component", "cfc.users_service").init()>

Objects are constructed per request. Existing singleton metadata in components is only descriptive and has no runtime effect in ColdFusion.

BaseDAO Utilities

  • executeQueryWithRetry(sql, params, options) retries transient connection errors with backoff.
  • queryToArray(query) normalizes ColdFusion query results into uppercase-key structs.

Application Scope

Application.cfc loads API credentials into application scope on startup. The API sub-application in api/v1/Application.cfc disables sessions and converts unhandled failures into JSON 500 responses. The root application also initializes a separate UserReview auth service and routes /UserReview/ requests through a distinct request context.

14. Deployment and Security

Required Configuration

  • Set OS-level environment variables UH_API_TOKEN and UH_API_SECRET.
  • Configure the ColdFusion datasource UHCO_Directory.
  • Install the IIS rewrite rule that routes /api/v1/* to /api/v1/index.cfm.
  • Schedule UH Sync and Data Quality tasks through the ColdFusion Scheduler.
  • Run the UserReview schema and permission seed scripts, including sql/create_user_review.sql and sql/add_user_review_permissions.sql.
  • Grant settings.user_review.manage and/or users.approve_user_review to the appropriate admin roles or direct admin users.

Compatibility Notes

  • Developed on ColdFusion 2025 and compatible with ColdFusion 2021.
  • queryExecute return type support is available in CF2021.
  • Undocumented fetchSize options are ignored safely in CF2021.
  • arraySort() closure callbacks and qry.getRow(1) are supported on the deployed version line.

Security Follow-Ups

  • Remove hardcoded fallback API credentials.
  • Restrict admin/ with proper authentication controls.
  • Lock down admin/diagnostic.cfm to admins only.
  • Enforce HTTPS for all admin and API traffic.
  • Review token IP allowlists and rotate UH API credentials periodically.
